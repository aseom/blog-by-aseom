]]> No ‘Access-Control-Allow-Origin’ 해결 http://blog.aseom.net/1351 Thu, 07 Jan 2016 15:52:08 +0000 http://blog.aseom.net/?p=1351 [계속 읽기...]]]> 최근의 웹 브라우저들은,
도메인이 다른 서버의 데이터를 가져오는 경우.

예를 들면, http://aseom.net/index.html에서
http://lib.aseom.net/foobar.css를 link 태그로 불러오면

No 'Access-Control-Allow-Origin' header is present on the requested resource.

와 같은 에러를 띄우면서 해당 리소스를 차단한다.

보안상의 이유인데, 자세한 설명은 다른 분의 글로 대신하기로.

아무튼, IIS 기준으로 이 문제를 해결하는 방법은 다음과 같다.

필자의 경우 lib.aseom.net에서 웹폰트를 불러와야 했는데,
해당 서브도메인을 가진 사이트를 선택하고, HTTP 응답 헤더 설정으로 이동,
추가를 누르고 ‘Access-Control-Allow-Origin’의 값을 ‘*’로 하면 끝.

값을 그냥 와일드카드(*)로 하기 보다는,
‘http://aseom.net/*’ 이런식으로 하는 걸 권장.

‘http://aseom.net/’에서 리소스를 가져가는(?) 것을 허용한다는 의미다.

]]> 서버 백업을 위한 Batch 스크립트 http://blog.aseom.net/1333 Thu, 07 Jan 2016 06:41:51 +0000 http://blog.aseom.net/?p=1333 [계속 읽기...]]]> 필자의 컴퓨터에는 마인크래프트 Spigot 서버와 IIS 웹서버가 구동되고 있다.

하루에도 몇번씩 커뮤니티 사이트에 랜섬웨어 피해가 올라오는 이 혼돈의 상황 속에서,
마인크래프트 월드 파일과 블로그 글들이 날아가는 건 상상조차 하기 싫은 일.
무조건 백업해야 한다.

외장하드든 나스든 랜섬에는 얄짤없으므로, 선택할 수 있는 건 클라우드 서비스였고,
매달 100GB를 $1.99에 제공하는 구글 드라이브를 백업용으로 쓰기로 했다.
만약 암호화된 파일이 동기화 되어버려도 30일 안에는 이전 버전 복구 가능.

백업 스크립트 (윈도우)

매일 수동 백업은 귀찮은 일이므로, 스크립트로 처리하는 것이 편하다.
커맨드라인을 지원하는 7-Zip으로 백업할 폴더를 통째 압축하도록 하는데,
한가지 문제가 발생. 서버 구동 중에 이렇게 하면 엑세스 오류가 난다.

다른 프로세스(Spigot 서버)가 사용 중인 파일에 7-Zip이 접근하지 못하는 문제인데,
윈도우에 포함된 볼륨 스냅샷 서비스(VSS)를 이용해서 볼륨의 스냅샷을 생성한 다음 백업을 진행하면 문제가 발생하지 않는다.
자세한 지식은 없어서 구글링해본 결과
VSS를 이용한 폴더 복사를 지원하는 HoboCopy라는 유틸을 발견했다.

hobocopy /full /r /skipdenied <원본 폴더> <복사할 폴더>
  • ‘/r’ 스위치가 있어야 폴더 내의 폴더까지 복사됨.

복사가 끝나면, 7-Zip으로 압축해서 백업 디렉토리에 저장하면 되겠다.

7z a -t7z <생성할 압축파일.7z> <압축할 폴더>
  • ‘-sdel’ 스위치를 넣으면 압축 후 폴더를 삭제한다.

이제 MySQL 데이터베이스 덤프. (7z 형식으로 압축)

mysqldump -u<유저> -p<패스워드> --databases <DB명> | 7z a -t7z -siSQL_Dump.sql <생성할 압축파일.7z>

몇달 전, JavaScript를 이용해서 Windows 10 앱을 만들 수 있다는 소식을 접했습니다.
부족한 개발 실력이지만 일단 부딪쳐 보자는 생각으로 클리앙 앱 개발을 시작했고,
좀 시간이 걸렸지만, ‘자작 앱을 만들어서 스토어에 올리자’는 목표는 이뤄졌죠.

그 때부터 지금까지를 돌아보니
서버, 웹 개발, 버킷 플러그인 개발, Win10 앱 개발, etc…
하고싶은 게 너무 많아서, 엄청나게 다양한 일을 벌여왔던 것 같습니다.

점점 머릿속이 복잡해지고,
문득 정리가 필요하겠다는 위기감(?!)에 앞으로의 계획을 세워봤습니다.
무언가를 만들고 나면 또 다른 걸 하느라
유지 보수는 소홀히 하게되는 것을 극복하고 싶었으니까요.

장기적으로 진행할 몇개의 길을 정하고, 그 안에서 하나하나에 전념하려고 합니다.
오래 지속될 수 있다고 장담 못하는 것들은 과감히 떨치자는 의미이기도 합니다.

이런 저런 고민 끝에 장기 진행으로 선택되지 못한
‘Clien 10’의 개발은 이쯤에서 제쳐두고,
나머지 것들에 집중하기 위한 양해를 구하려 이 글을 쓰게 되었습니다.
업데이트는 못하더라도, 스토어에서 앱이 내려가진 않을 듯 하네요.
